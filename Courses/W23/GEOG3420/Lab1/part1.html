Part 1: Introduction to Geoprocessing

What is geoprocessing?

In this set of lab exercises, we will be analyzing satellite imagery and other types of Earth Observation (EO) data by writing Python-based geoprocessing scripts. The term geoprocessing is used to describe any of the various processes in which specialized software tools are used to compare, analyze, or manipulate map layers, and their underlying geospatial data, to create new data. In past courses, you may have become familiar with using graphical user interface (GUI) based software, such as ArcGIS/Pro and QGIS, to manipulate geospatial data. Typically, when using this type of software, you open a specific tool's dialog, enter the appropriate parameters, which may include input/output files and various settings, and then display the results. You may have even used this approach to carry out sophisticated geospatial analysis workflows, involving several operations carried out in sequence. This essentially describes manual geoprocessing. The problem with this approach is that it doesn't scale well to real-life problems, i.e. the types of tasks that geospatial professionals encounter day-to-day work. For example, it may be the case that you need to perform the same geoprocessing workflow on dozens, hundreds, or even thousands of different input data sets, or perhaps you need to repeat the same process periodically in the future. When this is the case, you will likely need to adopt an automated geoprocessing approach.

Perhaps you've already performed some automated geoprocessing in previous courses, maybe in the form of an ArcGIS/Pro ModelBuilder or other graphical modeller, or perhaps even a Python or R script. Scripting (i.e. programing) is a particularly flexible and extremely powerful means of developing automated geoprocessing workflows. This is the reason why so many job advertisments in the fields of Remote Sensing and GIS ask for programming experience from applicants. Python programming experience in particular is among the most common requests for employers in geomatics.

WbW for geoprocessing

We'll be using the Whitebox Workflows for Python (WbW) software for automated geoprocessing in this course. WbW is a Python library for geoprocessing and includes hundreds of functions for manipulating and analyzing geospatial data. WbW is built on top of the WhiteboxTools open-source geospatial analysis platform. However, it is quite different from WhiteboxTools and has certain characteristics that make it better suited to the types of analysis that we'll be carrying out in these exercises. Most importantly, it allows us to directly manipulate geospatial data objects in our Python scripts.

Installing WbW

You'll need to install WbW on your computer to use it. Now that you have confirmed that you have Python installed on your computer, you should be able to simply type the following line at the command prompt of your terminal application to install WbW: 

pip install whitebox-workflows

Use pip3 if Python 2.X is your default system Python version. pip is Python's default package manager for installing and updating various Python libraries.

If you have an older version of WbW installed and want to update it to the latest version, simply type the following command:

pip install whitebox-workflows -U

For a demonstration of the install process, please see this YouTube video.

Using WbW

Now that we've ensured that Python is available and that WbW is installed on our computer, it's time to write our first Python WbW script. To do this, we need to use a Python code editor. In reality, we could use any text editor (e.g. Notepad), however, there are benifits in using software that is specifically designed for writing code. Most notably, autocomplete is a powerful function that greatly simplifies the coding experience. For our purposes, I will suggest that we use Visual Studio Code for our scripting needs, although you can use whatever coding environment you are most comfortable with (e.g., Python's default IDLE). If you're on your own computer and don't have VS Code installed, do so for free by following the instructions on this site.

Open VS Code and, using the File Explorer on the left panel, create a new file called wbw_text.py in a directory that you will dedicate for this course (it may be on a network drive or a USB stick, but if it is on the local C-drive of one of the lab computers, be sure to back-up your data/code each time before logging out of the computer). Either copy and paste or type the following code into this newly created file.

wbw_test.py

import whitebox_workflows as wbw

wbe = wbw.WbEnvironment('your-license-id')
try:
    # For now, let's just print the WbW version to make sure things are set up correctly.
    print(wbe.version())

    # But this is where we would put calls to various geoprocessing functions.
except Exception as e:
  print("The error raised is: ", e)
finally:
    print(wbe.check_in_license('your-license-id')) # print to confirm the successful check-in

Note: Python is a space-sensitive programming language and you need to have the indentations absolutely correct in your script for it to run correctly. This is something that can often go wrong when you are copying and pasting code into a script file.

Before trying to run this script replace the your-license-id in both places in the script above with the specific license ID string that your TA provided you in lab. You can run your script either by pressing the triangle-shaped 'Run-code' button in the menu (this may requre that the Code Runner extension is installed first), or simply by running the script in the terminal. VS Code has a built-in terminal usually located on the bottom of the main panel. To run your script from the terminal, cd (chdir in the Windows command prompt), i.e., change your directory, to the location of your script file and then simply type python wbw_text.py.

After the script has run, if Python and WbW have been correctly installed, you should see something similar to this:

[Running] python -u "/Users/jl/Documents/teaching/GEOG3420/Labs/Lab1/Code/wbw_test.py"
Whitebox Workflows for Python v1.0.7 by Whitebox Geospatial Inc.
Developed by Dr. John B. Lindsay, (c) 2022-2023

Description:
Whitebox Workflows for Python is an advanced geospatial data analysis platform
and Python extension module.
Successful license check-in

[Done] exited with code=0 in 0.52 seconds

There are some important things to note about our script above. We use a floating-license ID to unlock the WbW library. When the line wbe = wbw.WbEnvironment('your-license-id') is executed, WbW communicates with a license server hosted on a cloud service (note, this requires an Internet connection), and asks whether there is an available floating seat in the pool of seats associated with our floating-license ID. If there is, it will check one seat out and the script will continue to run, in this case, printing the versioning information for WbW. If no licence seat is available, the script will report the error, "All floating licenses are currently in use".

A important note on checking-in your license: Because the floating license uses a check-out/check-in licensing model, you must remember to check-in your license each time at the end of your script. That's why we have placed the check-in statement print(wbe.check_in_license('your-license-id')) in a finally block. This way, if our script throws an error during the processing, the check-in should still occur at the end.

I have set up enough floating-license seats for WbW such that everyone in our class should be able to run their scripts simultaneously. However, if you don't remember to check-in a license, or if our script throws a panic error (e.g., if we input an improper parameter value to a function), then other students in the lab may find that there is no seat available when they need to run their own script. You don't wnat to be that student who causes everyone's work to grind to a halt, so always be sure to check-in your license! In the event that you do end up with an un-checked-in license, there are ways to deal with that situation. See the paragraph starting 'Sometimes a script may panic...' in the WbW user manual for information about how to handle 'zombie' licenses.
